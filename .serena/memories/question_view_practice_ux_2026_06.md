# Question view and Practice Mode UX decisions from 2026-06 work

- In the desktop question view, metadata should be compact and side-by-side where possible: Topic, Concept, Cognitive Stage. On mobile it can stack.
- The answer-choice area should have a subtle divider above it. The prompt (`Your choice`/localized label) and horizontal radio buttons should be centered relative to the question frame, not the viewport edge. Radio spacing was increased for scanability.
- On question changes, the question view should scroll back to the top. Track the actually rendered `frage_idx`, not only `_current_question_idx`, because some navigation paths set `_current_question_idx` before render.
- In Practice Mode, after answer submission and immediate feedback, keep the Next button fixed at the bottom of the viewport and horizontally centered over the visible main question column, not the full viewport when the sidebar is open. Still scroll to the Next-button anchor after answering so feedback starts above the fixed button, and keep bottom spacing after visible feedback so explanations cannot be hidden behind it.
- Streamlit top chrome/header can cover elements if the target offset is too small. Use about `96px` top offset for Next-button and question-expander scroll targets; `24px` was too high and caused clipped titles/buttons.
- Opening question expanders should pull the expander heading upward with a DOM-side toggle listener scoped to the question view. Limit this behavior to the question view so start page, sidebar, and summary expanders are not unexpectedly moved.
- The detailed-explanation button is a special case: it triggers a Streamlit rerun and opens an expander already expanded, so set a one-shot session flag and scroll to an anchor before the expander after rerender.
- Practice feedback labels should be consistent and calm: render `Deine Antwort`/`Richtig`/`Erklärung` (localized equivalents) through one small label helper/CSS pattern, strip trailing translation colons for display, and avoid scattered inline styles.
- The standard explanation block after an answer should stay unframed in the lighter Practice Flow: use a subtle divider plus the explanation label, not a bordered container.
- Streamlit alert and expander labels can look vertically off because markdown paragraphs keep bottom margins and alert markdown containers can carry negative margins. In the question view, center `stAlertContainer`/`stAlertContent*`, reset alert paragraph margins, and reset expander-summary markdown/paragraph margins; keep the material arrow icon inline-flex and centered.
- Browser verification for these alignment fixes should inspect computed styles and center deltas, not only visible text. A good target is `labelCenterDelta = 0` and `iconCenterDelta = 0` for expander summaries; alert paragraph and alert container centers should align.
- When validating injected Streamlit CSS, do not trust an already running session blindly. If the expected rule is missing from DOM `<style>` tags, restart the local Streamlit server and reload before judging the visual result.
- Answer-option highlighting uses the separate rendered option labels, not only Streamlit's radio labels. Mirror the radio selection into the rendered option badge/text on the client side so the selected option highlights immediately before submit; do not rely on a submit rerun for the highlight.
- Streamlit forms must always render at least one `st.form_submit_button`. If an answered/locked question should not accept answers, render the radio/options outside a form and skip the form entirely instead of conditionally hiding all submit buttons inside the form.
- In the question view, Bookmark/Skip controls sit above the answer submit controls. Keep Bookmark/Skip outside the answer form so they remain independent actions and do not require an answer submit.
- For answer submission, avoid doing `st.rerun()` or heavy state transitions while still inside a `with st.form(...)` render path. Queue answer actions in `st.session_state` via `on_click`, then process the queue at the beginning of the next render before drawing answer controls; this prevents transient submit buttons or Streamlit form warnings on the feedback page.
- Cooldown and "select an answer first" feedback should remain `st.toast` messages, not inline warnings below submit buttons. Trigger toasts outside the form context.
- The feedback page has its own Bookmark control. Do not render a second Bookmark toggle for the same question there with a different widget key, because two toggles can fight over session state and undo each other.
- Bookmark-review navigation should keep showing the "Zur aktuellen Frage" return button even if the reviewed question is toggled off as a bookmark. The return target should prefer the active/current unanswered question over the reviewed feedback question itself.
- Panic Mode only skips waiting/cooldown constraints. It must not unlock already answered questions: radio choices, submit buttons, and skip controls stay disabled/hidden for answered questions even when Panic Mode is active.
